Why SQLite degrades

One grant insert becomes many scattered page writes

One logical row into insert six B-tree inserts, and each must find the leaf page its key belongs on. Our grant index keys are opaque IDs, so those leaves are scattered across the file.

SQLite write cost1 row → 6 B-tree inserts → scattered pages
Why SQLite updates hurt

Most updates rewrite a marshaled protobuf blob

SQLite cannot update one field in place because our common path stores the rich object in data BLOB.

SQLite update costrow read → protobuf decode → protobuf encode → row rewrite
SQLite compaction

SQLite compaction is an attached-database merge

The compacted output is a new SQLite DB. Each input c1z is attached, merged with SQL, then the output DB is saved as a v1 c1z.

SQLite compaction sends the working dataset through SQLite: rows merge into a new SQLite DB, and SQLite maintains the table and index B-trees.
The newer check is row metadata: external_id, sync_id, and discovered_at; no proto unmarshal is needed for the winner rule.
The final save still compresses the entire output SQLite DB, even when the partial was tiny.

Pebble strategy 1 · K-way

K-Way Merge: bucketed, streaming, recursive merge sort

Chosen when we can't do anything else.
secondary indexes

Primary winners also produce index SSTs

For each winning primary record, K-way shallow-scans the raw protobuf bytes to derive secondary index keys.
Index keys sort differently from primary keys, so each index family is written to its own run file via indexWriterSet.
Those run files are externally sorted in bounded chunks, k-way merged into per-index SSTs, and ingested with the primary SST.
K-way strategyMergeFilesInto · indexWriterSet
Pebble strategy 2 · Overlay

Overlay: a compactionless rebuilding scan

Chosen when we can't fold or must compact.
secondary indexes

Winner records also produce the output indexes

Normal Overlay writes derived index keys inline with the primary winner batches.
For a huge base bucket, avoid re-deriving every base index key. Copy the base's existing index keyspace, filtered by the seen set, and ingest the remaining keys as an index SST.
A bucket spills into to K-way if it grows too large; K-way rebuilds the index SSTs from the primary winners.
Overlay strategyMergeFilesIntoOverlay
Pebble strategy 3 · Fold

Fold: replay changes only

Fold saves big on compression by reusing unchanged SSTable frames. Overwritten records become fold_dead_bytes. Repeated folds grow output until a rebuild reclaims them.

Chosen when there is a base and the partials are small — summed partial file sizes are at most ~10% of the base, and the base's accumulated fold waste is still under the cutover.
Fold strategycompactPebbleFold / MergeInto
Fold save behavior

Fold splices unchanged SSTable frames

Fold keeps base SSTables byte-identical, so saving the compacted c1z can copy unchanged payload frames and only re-encode changed SSTables.

Why this matters: current c1z files can be 100+ GB, and compressing that much data is expensive even before any merge work.
Trade-off: overridden records leave dead bytes inside the spliced frames until a rebuild reclaims them.
Fold save behaviorWriteEnvelopeWithReuse
Pebble strategy selection

Pick the cheapest strategy for the input shape

Strategy selection uses cheap file metadata: partials vs. base size, plus accumulated fold dead bytes. No payloads are unpacked just to choose a plan.

1 · try first

Fold if possible

Use fold when there is a base, partials are tiny next to it, and accumulated fold dead bytes are still under the cutover.

2 · fallback

Overlay if not fold

If partials are too large, inputs are evenly sized, or fold dead bytes exceed the cutover, build a fresh output with Overlay.

3 · last resort

K-way if not overlay

If any record bucket is too large for Overlay's seen set, that bucket spills to bounded-memory K-way.

Selection is a fallback chain: Fold for the common big-base/tiny-partial shape; Overlay when fold does not fit or got too wasteful; then K-way only for buckets Overlay cannot bound in memory.

Separate effort · Grant expansion

Grant Expansion and you

The C1 data model ordinarily requires we fully materialize implicit grants - ie, a member of a group in Okta is granted access to some App. Expansion often takes much longer than syncing the upstream data (10+ hours). In these cases, we may explode ~7 million grants into over 100 million easily dwarfing all other records in a sync.

X · not compaction This is a separate sync-phase problem: grant expansion computes access implied by membership; compaction merges c1z sync history.
Expandability is stored next to the grant: the expansion blob + needs_expansion flag (Pebble: GrantExpandableRecord + a by_needs_expansion index).
Expansion runs as a sync phase after grants are synced, and writes new GrantImmutable grants that record their provenance.
Grant expansionexpandable grants → transitive closure
Expansion graph

Entitlements form a DAG; cycles collapse to one node

The graph is relatively small: no more than tens of thousands of entitlement nodes. The grant rows attached to that graph are the large part: millions of principals and memberships flowing through those edges.

A node is a set of entitlement IDs. Most start as a single entitlement and only grow when a cycle merges several into one node.
The algorithmic trick is to plan over the smaller entitlement DAG, then stream/merge the much larger grant sets behind each node.
Strongly-connected components are collapsed (FW–BW condensation): the super-node holds the union of its entitlements, and members of a cycle do not propagate to each other.
After collapse the graph is acyclic, so destinations can be reduced once every parent is finalized.

Classic expander · SQLite

Source-batched, layer by layer

The original evaluator walks expandable sources in depth layers, reads each source's grants, and fans them out to a batch of destinations. It is correct, but it re-does work on busy graphs.

Types: sourceEntitlement and destEntitlement are graph nodes/edges; sourceGrant, descendantGrant, and derivedGrant are grant rows.
# entitlement graph loop, then grant-row loop for sourceEntitlement in sources_whose_ancestors_are_expanded(graph): sourceGrants = ListGrantsForEntitlement(sourceEntitlement) # []Grant rows, paged destEntitlements = destinations(sourceEntitlement) # []Entitlement edges, small batch for destEntitlement in destEntitlements: existing[destEntitlement] = prefetch_principals(destEntitlement) for sourceGrant in sourceGrants: # one principal's grant on sourceEntitlement for destEntitlement in destEntitlements: # ← |sourceGrants| × |destEntitlements| if destEntitlement.shallow and not sourceGrant.is_direct: continue descendantGrant = existing[destEntitlement].get(sourceGrant.principal) if descendantGrant is UNKNOWN: # oracle incomplete (page cap) → descendantGrant = GET existing grant for (destEntitlement, sourceGrant.principal) # RANDOM GET if descendantGrant: merge sourceEntitlement into descendantGrant.sources else: emit derivedGrant(destEntitlement, sourceGrant.principal) flush(newGrantRows)

Classic expander · cost

Fan-in plus random GETs dominated

Classic batching reduced repeated source reads, but perf is dominated by other access patterns. Sources repeatedly update the same destination grants (read-modify-writes) and point lookups in the innermost loop tank performance.

Massive Proto churn: rewriting a destination once per source means its grant is unmarshaled and re-marshaled every time — repeated protobuf encode/decode that the single-pass merge avoids entirely.
A random GET in the innermost loop: in the grant × dest body, a per-principal point lookup (listDescendantGrantsForPrincipal) checks for an existing grant. Random reads thrash the cache.
How many: when the prefetch oracle misses (page cap exceeded), it's one GET per (source grant × destination) — up to |grants| × |dests| per source, and O(Σ grants × fan-out) across the graph. The prefetch oracle hides it only when it fits in memory.
The fix: the topological evaluator merges principal-ordered streams — the principal's grants are already in hand, so there is no per-principal GET at all.
Classic expander · costfan-in rewrites dominate; fan-out is batched
Topological projection · Pebble

Remove the hot paths

repeated source reads

Temporary secondary index

High-reuse source entitlements are scanned once into a scratch Pebble index keyed as (source, principal, grant) (value = direct? + principal proto), so their parent grants aren’t re-materialized through by_entitlement for every destination that shares them.

inner-loop GET + rewrites

K-way streaming merge

Parent streams and the destination’s own grants arrive in principal order and merge once per destination. The base grants are just another stream — so there is no per-principal GET, and no per-source rewrites or protobuf churn.

# reduce each destination once; merge parent streams by principal plan = build_expansion_plan(graph) # pick high-reuse sources by fan-out and grant volume project(plan.hot_sources) # scan once → sorted SST ingest → temp secondary index for node in topological_order(graph): # parents finalized before children for dest in node.entitlements: streams = [ grants_on(dest) ] # base, in principal order for source in parents(dest): # projected index or live store streams += parent_grants(source) # k-way merge by principal — no per-principal GET for principal, heads in kway_merge(streams): contrib = {} # map[source]→direct for h in heads: contrib = combine(contrib, h) # union; direct wins upsert derived_grant(dest, principal, contrib) flush_dirty()
K-way merge sort: the parents' streams and the destination's base grants arrive in principal order (the by_entitlement index), so a heap merges them in one pass — the base stream replaces the old per-principal listDescendantGrantsForPrincipal GET.
Temporary secondary index: high-reuse source entitlements are chosen by graph shape, scanned once, and bulk-ingested into a scratch Pebble DB. Key = (source_entitlement, principal_rt, principal_id, grant_id); value = direct? + principal proto.
Writes: winners go through PutExpandedGrantRecords (batched keep-newer, unsynced), which maintains the canonical secondary indexes.

Confidence

Why we trust the new evaluators

A different engine runs a different algorithm, so the real question is: does it produce the same access? We have't formally proven it, but layered testing gives high confidence the Pebble evaluators match the classic SQLite result.

ground truth

Byte-level parity vs classic SQLite

The classic SQLite expander is the reference. Every new evaluator (streaming & projection, on both engines) must match it grant-for-grant: id, principal, and sources with their direct flags.
randomized

Differential testing on random graphs

Random graphs and random stores are expanded both ways and diffed across many seeds — the cases nobody thought to write by hand.
fuzz

Two SCC fuzzers

FuzzCondenseFWBW_FromBytes throws arbitrary graphs at cycle condensation; FuzzCondenseFWBW_Cancellation cancels mid-run — correctness and crash-safety under hostile input.
stress & resume

Cycles, resume, and whale-scale

Interrupt-then-resume must reproduce the same output; cycles have a dedicated suite; and whale-scale graphs exercise the real shapes at size.
provenance semiring

Order-independent contribution merge

map[source_entitlement]→isDirect combines by union; direct membership wins over indirect.
This is the same shape as Green, Karvounarakis, and Tannen's 2006 provenance semiring work for Datalog.

Beyond the algorithms

The other half: allocations, pools, and cheaper writes

The new algorithms set the ceiling; a lot of the realized speedup came from mechanical engineering that cut allocations and reused work — without changing what's computed.

fewer allocations

~90% fewer allocs in the k-way merge

Reuse scratch buffers per record instead of allocating fresh key/blob slices every time.
Stop run-record header arrays from escaping to the heap through io.Reader/io.Writer.
Drop the per-index-key heap slice; sort views into one arena.
Borrow values from the iterator instead of copying key + value per record.
pooling & reuse

Don't rebuild expensive objects

Pool zstd encoders and decoders so each compress/decompress doesn't mint a new one.
A per-compaction envelope decoder pool, so opening each source reuses one decoder.
Hoist bound-method closures out of the record loops.
cheaper writes

Less work per grant written

Don't unmarshal protos unless the code actually needs object fields; keep hot paths on raw records / indexed side columns.
Deterministic grant IDs built without fmt.Sprintf; one shared GrantImmutable annotation.
PutExpandedGrantRecords: single read-before-write, in-batch dedup, arena-backed keys.
Unsynced (NoSync) batch commits — durability is restored once at sync end.
slim data models

Store IDs, not nested protos

Grant rows carry entitlement/principal IDs in indexed columns or v3 key fields, so hot paths do not need full nested protos.
Slim grant blobs strip embedded entitlement/principal objects and hydrate stubs only when the reader API needs them.
Expansion metadata is stored in side columns / records, shrinking what must be read, copied, and re-written.

What's next

The future

Where the storage-engine and expansion work goes from here.

1 · c1z replay

General c1z partial replays

Generalize replay beyond expansion: carry forward any .c1z records (resources, entitlements, grants) whose inputs are unchanged.
Skip re-syncing and re-storing untouched subsets — replay them from the prior sync and only do work for what actually moved.
Rewrite Entra, GitHub, GCP, and others.
2 · tuning

Tune fan-out vs fan-in

Optimize the fan-out (projection) path further — it is the less-exercised case.
Plot the projection-vs-streaming cost curve so the thresholds come from data, not a hardcoded fan-out ≥ 16.
3 · scale

Cross-layer fan-in & parallelism

Investigate reducing fan-in across layers instead of strictly level by level.
Expand independent destinations / buckets in parallel where the graph allows it.
4 · incremental

Partial grant expansion

Drive expansion from the event feed (and/or partial sync).
Re-expand only the changed subgraph, not the whole graph — the natural next multiplier on top of incremental sync.
5 · expansion replay

Grant replays on graph invalidation

Reuse a previous sync's expanded grants instead of recomputing them.
Invalidate only the parts of the entitlement graph that changed and replay the rest from the prior expansion output.
6 · remote diffs

Split c1z frames in S3

Store c1z payload frames as separately addressable S3 objects, keyed by their stable frame identity.
Upload and download only changed frames instead of paying the full c1z transfer cost for every diff.
